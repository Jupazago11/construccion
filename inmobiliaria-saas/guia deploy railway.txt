GUIA PARA DESPLEGAR PROYECTOS EN RAILWAY
========================================

Objetivo
--------
Esta guia sirve como checklist para preparar un proyecto antes de subirlo a Railway,
especialmente proyectos Laravel, Node, apps con Docker, monorepos y servicios con
PostgreSQL.


1. Revisar la estructura del repositorio
---------------------------------------
Antes de desplegar, confirmar donde vive realmente la app.

Ejemplos:

Caso simple:

repo/
  composer.json
  artisan
  package.json

Railway puede construir desde la raiz.

Caso monorepo:

repo/
  backend/
  frontend/
  app-laravel/
    composer.json
    artisan
    package.json

En Railway hay que configurar Root Directory con la carpeta real de la app.

Ejemplo:

Root Directory: app-laravel

Si Railway analiza solo:

./
└── app-laravel/

y dice que no puede detectar el proyecto, casi siempre falta configurar Root
Directory.


2. Variables necesarias en Railway
----------------------------------
Las variables deben configurarse en el servicio de la APP, no solo en el servicio
de base de datos.

Para Laravel con PostgreSQL:

APP_NAME="Nombre App"
APP_ENV=production
APP_DEBUG=false
APP_URL=https://tu-dominio.up.railway.app
APP_KEY=base64:...
APP_LOCALE=es
APP_FALLBACK_LOCALE=es

SUPERADMIN_USERNAME=superadmin
SUPERADMIN_NAME="Super Admin"
SUPERADMIN_EMAIL=admin@example.com
SUPERADMIN_PASSWORD=clave-segura
# Opcional, solo si se quiere resetear la clave en cada deploy:
# SUPERADMIN_RESET_PASSWORD=true

DB_CONNECTION=pgsql
DATABASE_URL=${{Postgres.DATABASE_URL}}
DB_URL=${{Postgres.DATABASE_URL}}
DB_SSLMODE=require

LOG_CHANNEL=stderr
LOG_LEVEL=info

CACHE_STORE=database
SESSION_DRIVER=database
SESSION_ENCRYPT=false
QUEUE_CONNECTION=database

IMPORTANTE:
Las variables como DATABASE_URL, PGHOST, PGUSER, PGPASSWORD que aparecen dentro
del servicio Postgres no se pasan solas al servicio de la app. Deben referenciarse
desde la app usando ${{Postgres.DATABASE_URL}}.

Si el servicio de base de datos no se llama Postgres, usar el nombre real.
Railway suele autocompletar cuando escribes ${{.

Tambien recordar:

- APP_KEY debe estar en el servicio de la app.
- APP_KEY no debe cambiarse despues de produccion salvo que se acepte invalidar
  cookies, sesiones y datos cifrados.
- APP_URL debe ser el dominio real publicado por Railway, no localhost.
- Si la app se ve sin estilos, revisar el codigo fuente: los assets de Vite
  deben salir con https://. Si salen con http://, falta APP_URL correcto o
  confiar en los proxy headers de Railway.
- No subir el archivo .env al repositorio. Las variables productivas van en
  Railway.


3. APP_KEY en Laravel
---------------------
Laravel necesita APP_KEY en produccion.

Para generar una:

php artisan key:generate --show

o con Sail:

./vendor/bin/sail artisan key:generate --show

Copiar el resultado en Railway:

APP_KEY=base64:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

Error tipico si falta:

The /login healthcheck endpoint returned HTTP 500 because Laravel cannot handle
encryption, sessions, or CSRF tokens without APP_KEY.

Solucion:
Agregar APP_KEY en el servicio de la app y hacer redeploy.


4. PostgreSQL: error comun con SQLite
-------------------------------------
Error tipico:

SQLSTATE[HY000]: General error: 1 near "ALTER": syntax error
Database: /var/www/html/database/database.sqlite

Significa que Laravel esta usando SQLite, no Postgres.

Solucion:

En el servicio de la app agregar:

DB_CONNECTION=pgsql
DATABASE_URL=${{Postgres.DATABASE_URL}}
DB_URL=${{Postgres.DATABASE_URL}}
DB_SSLMODE=require

Ademas, en config/database.php conviene permitir DATABASE_URL:

'url' => env('DB_URL', env('DATABASE_URL')),

en la conexion pgsql.


5. Build en Railway: Railpack, Nixpacks o Docker
------------------------------------------------
Railway puede construir con Railpack/Nixpacks automaticamente, pero en proyectos
Laravel con dependencias PHP, Node, Vite, extensiones o monorepo, a veces es mas
estable usar Dockerfile.

Sintomas:

"Railpack could not determine how to build the app"
"Script start.sh not found"
"Failed to build an image"

Posibles causas:

- Railway esta mirando la raiz equivocada del repo.
- No se configuro Root Directory.
- composer.json o package.json estan en una subcarpeta.
- El proyecto necesita extensiones PHP no detectadas.
- El start command no esta claro.

Soluciones:

- Configurar Root Directory.
- Agregar railway.json.
- Usar Dockerfile si el autodetect falla.


6. Ejemplo de railway.json
--------------------------
Para app con Dockerfile en la raiz:

{
    "$schema": "https://railway.com/railway.schema.json",
    "build": {
        "builder": "DOCKERFILE",
        "dockerfilePath": "Dockerfile"
    },
    "deploy": {
        "preDeployCommand": "sh ./railway/init-app.sh",
        "healthcheckPath": "/login",
        "healthcheckTimeout": 300,
        "restartPolicyType": "ON_FAILURE",
        "restartPolicyMaxRetries": 10
    }
}

Para app donde Railway entra directo al Laravel:

{
    "$schema": "https://railway.com/railway.schema.json",
    "build": {
        "builder": "RAILPACK",
        "buildCommand": "npm run build"
    },
    "deploy": {
        "preDeployCommand": "chmod +x ./railway/init-app.sh && sh ./railway/init-app.sh",
        "healthcheckPath": "/login",
        "healthcheckTimeout": 300,
        "restartPolicyType": "ON_FAILURE",
        "restartPolicyMaxRetries": 10
    }
}


7. Script de pre-deploy para Laravel
------------------------------------
Crear:

railway/init-app.sh

Contenido recomendado:

#!/usr/bin/env sh
set -e

if [ -z "${DATABASE_URL:-}" ] && [ -z "${DB_URL:-}" ]; then
    echo "Railway init error: configure DATABASE_URL or DB_URL with the Postgres service URL before deploying."
    echo "Example: DATABASE_URL=\${{Postgres.DATABASE_URL}} and DB_CONNECTION=pgsql"
    exit 1
fi

export DB_CONNECTION="${DB_CONNECTION:-pgsql}"
export DB_URL="${DB_URL:-${DATABASE_URL:-}}"

echo "Railway init: clearing cached Laravel files"
php artisan config:clear
php artisan route:clear
php artisan view:clear

echo "Railway init: running database migrations"
php artisan migrate --force

echo "Railway init: seeding production baseline"
php artisan db:seed --force

echo "Railway init: caching views"
php artisan view:cache

if [ "${RAILWAY_CACHE_CONFIG:-false}" = "true" ]; then
    echo "Railway init: caching config"
    php artisan config:cache
fi

if [ "${RAILWAY_CACHE_ROUTES:-false}" = "true" ]; then
    echo "Railway init: caching routes"
    php artisan route:cache
fi

echo "Railway init: done"

Nota:
No activar config:cache o route:cache al inicio si todavia se estan afinando
variables. Se pueden activar despues con:

RAILWAY_CACHE_CONFIG=true
RAILWAY_CACHE_ROUTES=true


8. Dockerfile para Laravel con Apache
-------------------------------------
Ejemplo base:

FROM php:8.4-apache

WORKDIR /var/www/html

RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        ca-certificates \
        curl \
        gettext-base \
        git \
        libfreetype6-dev \
        libjpeg62-turbo-dev \
        libonig-dev \
        libpng-dev \
        libpq-dev \
        libwebp-dev \
        libzip-dev \
        unzip \
    && curl -fsSL https://deb.nodesource.com/setup_24.x | bash - \
    && apt-get install -y --no-install-recommends nodejs \
    && docker-php-ext-configure gd --with-freetype --with-jpeg --with-webp \
    && docker-php-ext-install -j"$(nproc)" bcmath exif gd mbstring pcntl pdo_pgsql pgsql zip \
    && a2dismod -f mpm_event mpm_worker mpm_prefork \
    && rm -f /etc/apache2/mods-enabled/mpm_*.load /etc/apache2/mods-enabled/mpm_*.conf \
    && a2enmod mpm_prefork rewrite headers \
    && rm -rf /var/lib/apt/lists/*

COPY --from=composer:2 /usr/bin/composer /usr/bin/composer

COPY composer.json composer.lock ./
RUN composer install --no-dev --prefer-dist --no-interaction --no-progress --no-scripts --optimize-autoloader

COPY package.json package-lock.json ./
RUN npm ci

COPY . ./
RUN npm run build \
    && composer dump-autoload --optimize \
    && chown -R www-data:www-data storage bootstrap/cache \
    && chmod -R ug+rwX storage bootstrap/cache

CMD ["docker/start-container.sh"]

Si la app esta en una subcarpeta, ajustar COPY:

COPY app-laravel/composer.json app-laravel/composer.lock ./
COPY app-laravel/package.json app-laravel/package-lock.json ./
COPY app-laravel ./


9. Apache config para Laravel
-----------------------------
Crear:

docker/apache.conf

Contenido:

<VirtualHost *:${PORT}>
    ServerAdmin webmaster@localhost
    DocumentRoot /var/www/html/public

    <Directory /var/www/html/public>
        AllowOverride All
        Require all granted
        Options -Indexes +FollowSymLinks
    </Directory>

    ErrorLog ${APACHE_LOG_DIR}/error.log
    CustomLog ${APACHE_LOG_DIR}/access.log combined
</VirtualHost>


10. Start script para contenedor
--------------------------------
Crear:

docker/start-container.sh

Contenido:

#!/usr/bin/env sh
set -e

: "${PORT:=80}"

rm -f /etc/apache2/mods-enabled/mpm_*.load /etc/apache2/mods-enabled/mpm_*.conf
a2enmod mpm_prefork >/dev/null

sed -i "s/Listen 80/Listen ${PORT}/" /etc/apache2/ports.conf
envsubst '${PORT}' < /etc/apache2/sites-available/000-default.conf > /tmp/000-default.conf
cat /tmp/000-default.conf > /etc/apache2/sites-available/000-default.conf

mkdir -p storage/framework/cache storage/framework/sessions storage/framework/views bootstrap/cache
chown -R www-data:www-data storage bootstrap/cache
chmod -R ug+rwX storage bootstrap/cache

exec apache2-foreground

Nota sobre Apache MPM:
Si aparece "More than one MPM loaded", Apache esta cargando mas de un modulo MPM
al mismo tiempo. La limpieza de mpm_*.load y mpm_*.conf evita ese choque y deja
solo mpm_prefork habilitado.

-----------------------------
Crear .dockerignore:

**/storage/framework/sessions/*

**/.phpunit.result.cache


12. PHP version
---------------
Revisar composer.lock y composer.json.

Error tipico:

package requires php >=8.4 -> your php version (8.3.x) does not satisfy that requirement

Solucion:

- Usar una imagen PHP compatible, por ejemplo php:8.4-apache.
- Actualizar composer.json:

"php": "^8.4"

- Ejecutar:

composer update --lock

o con Sail:

./vendor/bin/sail composer update --lock --ignore-platform-req=php


13. Extensiones PHP comunes para Laravel
----------------------------------------
Segun el proyecto, puede necesitar:

- pdo_pgsql, pgsql: PostgreSQL
- gd: Excel, imagenes, thumbnails, algunos PDF
- zip: Excel y archivos
- mbstring: strings
- fileinfo: uploads y mime types
- bcmath: calculos
- exif: imagenes
- pcntl: workers/queues

En composer.json se pueden declarar:

"ext-fileinfo": "*",
"ext-gd": "*",
"ext-pdo_pgsql": "*",
"ext-pgsql": "*",
"ext-zip": "*"


14. Vite y assets
-----------------
Para Laravel con Vite:

npm ci
npm run build

El build debe generar:

public/build/manifest.json

Si aparece error de import:

Failed to resolve import "chart.js/auto"

Solucion:

npm install chart.js

Confirmar que package.json y package-lock.json esten actualizados y commiteados.


15. Storage externo: S3/R2
--------------------------
Para Cloudflare R2 o S3:

FILESYSTEM_DISK=r2
R2_ACCESS_KEY_ID=...
R2_SECRET_ACCESS_KEY=...
R2_DEFAULT_REGION=auto
R2_BUCKET=...
R2_ENDPOINT=...
R2_ROOT_PREFIX=nombre-del-proyecto
R2_USE_PATH_STYLE_ENDPOINT=true

Error tipico:

The PutObject operation requires non-empty parameter: Bucket

Solucion:

Falta R2_BUCKET o AWS_BUCKET en variables.


16. Logs en Railway
-------------------
Usar:

LOG_CHANNEL=stderr
LOG_LEVEL=info

Asi los errores aparecen en los logs de Railway, no solo en storage/logs.


17. Queue worker
----------------
Si el proyecto usa colas:

QUEUE_CONNECTION=database

Crear un servicio separado opcional con start command:

php artisan queue:work --sleep=3 --tries=3 --timeout=120

O script:

railway/run-worker.sh

#!/usr/bin/env sh
set -e

php artisan queue:work --sleep=3 --tries=3 --timeout=120


18. Scheduler / cron
--------------------
Si el proyecto usa tareas programadas:

Crear servicio separado con:

while [ true ]; do
    php artisan schedule:run --verbose --no-interaction
    sleep 60
done


19. Healthcheck
---------------
Para apps con login, usar una ruta publica:

healthcheckPath: "/login"

Evitar "/" si redirige raro, requiere auth, o depende de datos iniciales.


20. Errores frecuentes y solucion
---------------------------------

Error:
Railpack could not determine how to build the app.

Causa:
Railway mira la carpeta equivocada o el proyecto esta en subcarpeta.

Solucion:
Configurar Root Directory o usar Dockerfile.


Error:
Script start.sh not found

Causa:
Railpack espera un script de inicio y no lo encuentra.

Solucion:
Definir start command, railway.json o Dockerfile con CMD.


Error:
Failed to build an image. composer install exit code 2.

Causa:
Version de PHP incompatible o extension faltante.

Solucion:
Revisar composer output completo.
Usar PHP correcto, por ejemplo php:8.4-apache.
Instalar extensiones requeridas.


Error:
Laravel defaulted to SQLite.

Causa:
No se configuro DB_CONNECTION / DATABASE_URL en el servicio de la app.

Solucion:
En servicio app:

DB_CONNECTION=pgsql
DATABASE_URL=${{Postgres.DATABASE_URL}}
DB_URL=${{Postgres.DATABASE_URL}}


Error:
near "ALTER": syntax error, Database sqlite

Causa:
Migracion con SQL PostgreSQL corriendo sobre SQLite.

Solucion:
Configurar Postgres. Si se quiere blindar, hacer migraciones database-agnostic
con Schema Builder o condicional por driver.


Error:
APP_KEY missing

Causa:
No se configuro APP_KEY.

Solucion:
php artisan key:generate --show
Copiar en Railway.


Error:
No existe usuario inicial para entrar al sistema.

Causa:
Se corrieron migraciones, pero no se ejecuto el seeder de produccion o faltan
variables SUPERADMIN_*.

Solucion:
Dejar DatabaseSeeder solo con seeders base, por ejemplo:

RoleAndPermissionSeeder::class
SuperAdminSeeder::class

En Railway, despues de migrate --force ejecutar:

php artisan db:seed --force

Variables recomendadas:

SUPERADMIN_USERNAME=superadmin
SUPERADMIN_NAME="Super Admin"
SUPERADMIN_EMAIL=admin@example.com
SUPERADMIN_PASSWORD=clave-segura

Los seeders demo pueden quedarse en el repositorio, pero no deben estar llamados
desde DatabaseSeeder en produccion.


Error:
Healthcheck /login devuelve HTTP 500, pero Apache ya arranco.

Causa:
Laravel esta fallando en runtime. Las causas mas comunes son APP_KEY faltante,
APP_URL incorrecto, variables de base de datos incompletas, storage/cache sin
permisos, o config cache vieja.

Solucion:
Revisar deploy logs y runtime logs. Confirmar:

APP_KEY=base64:...
APP_ENV=production
APP_DEBUG=false
APP_URL=https://dominio-real
DB_CONNECTION=pgsql
DATABASE_URL=${{Postgres.DATABASE_URL}}
LOG_CHANNEL=stderr

Luego redeploy.


Error:
La pagina carga sin estilos, el logo se ve gigante o los inputs se ven como HTML
basico.

Causa:
El CSS/JS de Vite no esta cargando. En Railway suele ocurrir cuando Laravel no
confia en los headers del proxy y genera assets con http:// mientras el navegador
esta en https://. El navegador bloquea esos assets por contenido mixto.

Solucion:
Confirmar en el servicio app:

APP_URL=https://dominio-real.up.railway.app

Y en bootstrap/app.php confiar en los proxies de Railway:

$middleware->trustProxies(at: '*');

Despues hacer redeploy. Al ver el codigo fuente, los enlaces a /build/assets
deben iniciar con https://.


Error:
AH00534: apache2: Configuration error: More than one MPM loaded.

Causa:
La imagen Apache tiene mas de un modulo MPM habilitado, por ejemplo mpm_event y
mpm_prefork al mismo tiempo.

Solucion:
En Dockerfile y en el script de arranque limpiar los MPM y activar solo
mpm_prefork:

rm -f /etc/apache2/mods-enabled/mpm_*.load /etc/apache2/mods-enabled/mpm_*.conf
a2enmod mpm_prefork


Error:
AH00558: apache2: Could not reliably determine the server's fully qualified
domain name.

Causa:
Apache no tiene ServerName global.

Solucion:
Normalmente es solo una advertencia y no rompe el deploy. Si se quiere limpiar,
agregar ServerName localhost a la configuracion global de Apache.


Error:
Unable to write file at location... Bucket empty

Causa:
Variables R2/S3 incompletas.

Solucion:
Configurar R2_BUCKET, R2_ENDPOINT, keys y FILESYSTEM_DISK.


Error:
403 o imagen rota en archivos R2

Causa:
URL publica no configurada, bucket privado, o se esta intentando mostrar una URL
no firmada.

Solucion:
Usar URLs firmadas desde Storage::temporaryUrl si el bucket es privado, o
configurar public/custom domain si se desea publico.


21. Checklist final antes de deploy
-----------------------------------

[ ] La app real esta en la carpeta que Railway construye.
[ ] Root Directory esta bien configurado si es monorepo.
[ ] APP_KEY existe.
[ ] APP_ENV=production.
[ ] APP_DEBUG=false.
[ ] APP_URL apunta al dominio de Railway.
[ ] APP_LOCALE y APP_FALLBACK_LOCALE estan definidos si la app usa espanol.
[ ] SUPERADMIN_USERNAME, SUPERADMIN_EMAIL y SUPERADMIN_PASSWORD estan definidos.
[ ] DB_CONNECTION=pgsql.
[ ] DATABASE_URL apunta al servicio Postgres.
[ ] DB_URL tambien apunta al servicio Postgres, si Laravel lo usa.
[ ] Las variables de DB estan en el servicio de la app, no solo en Postgres.
[ ] LOG_CHANNEL=stderr.
[ ] npm run build funciona localmente.
[ ] composer install --no-dev funciona en entorno similar.
[ ] php artisan migrate --force funciona contra Postgres.
[ ] R2/S3 variables configuradas si hay uploads.
[ ] Healthcheck usa una ruta publica.
[ ] Apache no carga mas de un MPM si se usa php:apache.
[ ] No se sube .env al repo.


22. Comandos utiles de verificacion local
-----------------------------------------

Validar Composer:

composer validate --strict

Build frontend:

npm run build

Migrar con Sail:

./vendor/bin/sail artisan migrate --force

Probar build Docker:

docker build --progress=plain -t app-railway-test .

Verificar MPM de Apache dentro de la imagen:

docker run --rm --entrypoint sh app-railway-test -lc "ls -1 /etc/apache2/mods-enabled/mpm_* && apache2ctl -M | grep mpm"

Ver extensiones PHP:

php -m

Ver config de Laravel:

php artisan config:clear
php artisan about
